How It Works

A single webhook endpoint receives push events and triggers local pulls.

GitHub / GitLab push event
→
GitWH POST /wh
→
Validate repo & secret
→
git pull local repo(s)

Webhook Reception

The server listens for HTTP POST requests on the /wh endpoint.

Payload Detection

Automatically detects GitHub (application/x-www-form-urlencoded) or GitLab (application/json with X-Gitlab-Token) format.

Validation

Checks that the repository is configured and verifies the webhook secret if one is set.

Async Pull

Sends a pull event to a buffered channel. A background goroutine executes git pull with per-directory mutex locking and a configurable timeout.

Quick Start

Up and running in under five minutes.

Prerequisites

  • Go 1.20 or higher
  • Git installed on the system
  • Systemd (for service installation, optional)

1. Build

terminal
go install

This compiles and installs the binary into the default Go bin path. Alternatively:

terminal
go build -o /usr/local/bin/gitwh

2. Configure

Create a configuration file (YAML or JSON):

config.yaml
listen: ":8080"
buffer_size: 3
timeout: 10
repos:
  my-repo:
    secret: "your-webhook-secret"
    folders:
      - "/var/www/my-repo"

The repository key (my-repo) must match the actual repository name. For https://github.com/user/my-repo, use my-repo.

3. Run

terminal
# Default config location: /etc/gitwh.yaml
./gitwh

# Or specify a custom config
./gitwh -config /path/to/config.yaml

4. Install as a systemd service (optional)

terminal
./install.sh

The script copies the service file to /etc/systemd/system/, the config to /etc/gitwh.yaml, then enables and starts the service.

5. Set up webhooks

Point your GitHub or GitLab webhook to:

webhook URL
http://your-server:8080/wh

Configuration

Supports both YAML and JSON. The format is auto-detected by file extension.

Parameters

ParameterTypeDefaultDescription
listenstring:8080Address and port the server binds to
buffer_sizeint3Buffered channel capacity for concurrent events
timeoutint10Maximum seconds for a single git pull
reposmap—Map of repository names to their settings
repos.*.secretstring—Optional webhook secret for request validation
repos.*.folders[]string—Local paths to pull when a push is received

YAML example

config.yaml
listen: ":8080"
buffer_size: 3
timeout: 10
repos:
  gitwh:
    secret: "s3cret"
    folders:
      - "/opt/gitwh"
  website:
    folders:
      - "/var/www/html"
      - "/home/deploy/website"

JSON example

config.json
{
  "listen": ":8080",
  "buffer_size": 3,
  "timeout": 10,
  "repos": {
    "gitwh": {
      "secret": "s3cret",
      "folders": ["/opt/gitwh"]
    },
    "website": {
      "folders": ["/var/www/html", "/home/deploy/website"]
    }
  }
}

File extensions .yaml and .yml are parsed as YAML; .json and .conf are parsed as JSON.

Service Management

terminal
# Check status
sudo systemctl status gitwh.service

# Start / stop / restart
sudo systemctl start gitwh.service
sudo systemctl stop gitwh.service
sudo systemctl restart gitwh.service

# Follow logs
sudo journalctl -u gitwh.service -f

# Disable auto-start
sudo systemctl disable gitwh.service

Architecture

Clean package layout with a single responsibility per layer.

gitwh/
├── main.go — entry point, HTTP server setup
├── config/
│   └── config.go — load & parse YAML/JSON config
├── handlers/
│   └── handlers.go — chi router, webhook payload parsing
└── puller/
    ├── puller.go — Puller interface definition
    └── git/
       └── git.go — git pull with per-dir mutex & timeout

Security

Minimal surface, validated inputs, safe concurrency.

Secret Validation

Webhook secrets are checked before any pull is triggered.

Timeouts

Context-based timeouts prevent hanging git pull operations.

Mutex Locking

Per-directory locks prevent concurrent pulls on the same path.

Minimal Surface

A single POST /wh endpoint — nothing else is exposed.
